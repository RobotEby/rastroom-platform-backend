# Rastroom Platform Backend

A NestJS backend for Rastroom, built on the existing frontend. Customized order management and traceability: customers, orders, furniture, parts with QR codes, manufacturing processes, kit verification/assembly, shipping, dashboards, and notifications.

## Stack

- Node.js + TypeScript
- NestJS
- Prisma ORM
- PostgreSQL
- JWT access/refresh token
- Argon2 for passwords
- class-validator/class-transformer
- Swagger/OpenAPI at `/docs`
- Docker Compose for PostgreSQL
- Jest for testing
- ESLint

## Local Setup

```bash
cd rastroom-platform-backend
cp .env.example .env
docker compose up -d
npm install
npx prisma migrate dev
npm run prisma:seed
npm run dev
```

The API is accessible at `http://localhost:8081`.

## Variables

See `.env.example`. For local development:

```env
DATABASE_URL=postgresql://rastroom:rastroom@localhost:5432/rastroom?schema=public
PORT=8081
CORS_ORIGIN=http://localhost:8080,http://localhost:5173
```

## Main Modules

- `auth`: login, registration, refresh, logout, current user, and development password reset.
- `users`: profile, password change, user and role management.
- `clients`: client CRUD.
- `orders`: CRUD for orders and statuses.
- `furniture`: CRUD for furniture linked to orders.
- `parts`: CRUD for parts, QR codes, and batch import.
- `processes`: sequential steps, start/end, and execution logs.
- `assembly`: parent part search, child part checklist, and kit validation.
- `expedition`: ready orders and marking as shipped.
- `dashboard`: totals, parts per process, average times, recent activities, and alerts.
- `uploads`: local upload with metadata to the database.
- `notifications`: notifications generated by assembly/shipping.

## Database Entities

`User`, `Client`, `Order`, `Furniture`, `Part`, `Process`, `ExecutionLog`, `Notification`, `Upload`, `AuditLog`.

Main enums: `AppRole`, `OrderStatus`, `ProcessType`, `ProcessLogStatus`, `NotificationType`, `NotificationStatus`.

## Frontend

The frontend has been configured to use `VITE_API_URL=http://localhost:8081`, as specified in `rastroom-platform-frontend/.env.example`. It now stores JWTs in `localStorage` and calls the actual API for login, dashboards, listings, creation, editing, deletion, scanner, processes, assembly, and shipping.

## Decisions

- Field names follow the frontend (`client_id`, `estimated_delivery`, `current_process`, `finish_color_hex`) to avoid unnecessary adapters.
- Soft delete has been applied where it makes sense (`deleted_at`) for users, customers, orders, furniture, and parts.
- Uploads are local during development (`/uploads`) and the model leaves room for S3/external storage.
- Email notifications are stored in the database; actual sending can be integrated later.
- Error responses are standardized with `statusCode`, `message`, and `errors`.
